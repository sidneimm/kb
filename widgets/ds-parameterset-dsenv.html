📦 Parameter Set & DSenv

DataStage Clássico (Parallel Engine) — Gerenciamento de Parâmetros e Variáveis de Ambiente

O que são parâmetros no DataStage?

No DataStage clássico (v11.x / CP4D), parâmetros permitem externalizar valores que variam entre ambientes (dev, hml, prod) ou execuções — como strings de conexão, caminhos de arquivo, datas de referência e flags de controle.

Existem dois mecanismos principais:

1Parameter Set
Agrupamento de params
reutilizável em jobs
⟷
2DSenv
Variáveis de ambiente
do sistema operacional
Ambos podem ser usados juntos. Um Parameter Set pode referenciar variáveis DSenv, e o DSenv pode sobrescrever valores de um Parameter Set em tempo de execução.

Fluxo de resolução de um parâmetro em runtime

1. Job é invocado (manual, sequencer ou API)
↓
2. Engine carrega Parameter Set(s) associados ao job
↓
3. Valores do Parameter Set são lidos do arquivo .apt / valor default
↓
4. Variáveis DSenv sobrescrevem (se definidas no ambiente do processo)
↓
5. Valor final é usado pela stage em tempo de execução

O que é um Parameter Set?

Um Parameter Set é um objeto do DataStage Designer que agrupa um conjunto nomeado de parâmetros de job. Ele é criado uma vez no repositório e pode ser importado por múltiplos jobs, evitando duplicação.

Como referenciar no job

Após importar o Parameter Set no job, o valor é referenciado com a notação:

-- Notação: #NomeDoParametro#
-- Exemplos em Transformer / SQL Stage:

SELECT * FROM #DB_SCHEMA#.#DB_TABLE#
  WHERE dt_ref = '#DT_REFERENCIA#'

-- Em propriedades de stage (Source/Target):
Schema:  #DB_SCHEMA#
Table:   #DB_TABLE#
⚠️ Em Parallel Jobs, o delimitador é #PARAM#. Em Server Jobs (legado), usa-se %PARAM%. Não misture as notações.

Value File (.apt)

Um Parameter Set pode ter um ou mais Value Files associados — arquivos texto simples que fornecem os valores para um ambiente específico (dev/hml/prod). O arquivo é selecionado ao submeter o job ou via sequencer.

# Arquivo: /opt/IBM/InformationServer/Server/Projects/MeuProjeto/
#           ParameterSets/CONN_DW/prod.apt

DB_HOST=dw-prod.empresa.com.br
DB_PORT=5432
DB_SCHEMA=DW_PROD
DB_TABLE=FATO_VENDAS
BATCH_SIZE=50000
✅ Mantenha um arquivo .apt por ambiente. Nunca versione senhas em Value Files — use o tipo Encrypted ou referencie variáveis DSenv.

O que é DSenv?

O DSenv (DataStage Environment) é o arquivo de configuração de variáveis de ambiente lido pelo engine do DataStage no momento da inicialização de cada job. Funciona como um overlay do ambiente UNIX do processo.

Localização padrão

# IBM DataStage & QualityStage (instalação típica)
/opt/IBM/InformationServer/Server/DSEngine/dsenv

# Também referenciado pelo script:
/opt/IBM/InformationServer/Server/DSEngine/bin/dsenv
⚠️ O arquivo dsenv é lido pelo processo do DataStage Engine (uvsh). Alterações só têm efeito após reiniciar o engine ou abrir nova sessão de job.

Variáveis de usuário no DSenv

Você pode definir qualquer variável no dsenv e referenciá-la em parâmetros de job com a notação $NOME_VAR ou usando o tipo de parâmetro Environment Variable.

# /opt/IBM/InformationServer/Server/DSEngine/dsenv

# — Variáveis de sistema (não altere sem necessidade) —
export DSHOME="/opt/IBM/InformationServer/Server/DSEngine"
export APT_CONFIG_FILE="$DSHOME/configs/default.apt"

# — Variáveis de negócio (seu ambiente) —
export DS_ENV="PROD"
export DW_HOST="dw-prod.empresa.com.br"
export DW_PORT="5432"
export DW_SCHEMA="DW_PROD"
export ETL_WORKDIR="/data/etl/workdir"
export ETL_REJECT_DIR="/data/etl/rejects"

# — Ajustes de bibliotecas —
export LD_LIBRARY_PATH="/opt/oracle/lib:$LD_LIBRARY_PATH"
export ODBCINI="/opt/IBM/InformationServer/Server/DSEngine/odbc.ini"

Referenciar DSenv em um Parameter Set

No campo Default Value de um parâmetro, use a sintaxe:

# Parâmetro DB_HOST com default vindo do DSenv:
Default Value: $DW_HOST

# No Value File (.apt), também funciona:
DB_HOST=$DW_HOST
WORK_DIR=$ETL_WORKDIR/run_$DS_ENV
✅ Essa é a prática recomendada para senhas e hosts: armazene no dsenv (gerenciado pelo sysadmin/vault) e referencie no Parameter Set.

Ordem de precedência (quem ganha?)

Quando o mesmo parâmetro pode vir de múltiplas fontes, o DataStage segue esta hierarquia — do maior para o menor peso:

1
Override explícito na invocação do job dsjob -param NOME=VALOR · API REST · Sequencer com override
MAIOR
2
Value File (.apt) do Parameter Set Selecionado no job run / sequencer
ALTO
3
Variáveis DSenv (ambiente do processo) Definidas em /DSEngine/dsenv ou export na shell
MÉDIO
4
Valor Default do Parameter Set (no Designer) Fallback quando nenhuma outra fonte fornece o valor
BAIXO
⚠️ Atenção: em alguns contextos o DSenv pode ter precedência sobre o Value File dependendo de como o parâmetro é do tipo Environment Variable. Sempre teste o comportamento no seu ambiente.

Parameter Set × DSenv — quando usar cada um?

📦 Parameter Set

  • Parâmetros de negócio (schema, tabela, data)
  • Valores que variam por job ou por execução
  • Documentação visível no Designer
  • Suporte a Value Files por ambiente
  • Compartilhável entre múltiplos jobs
  • Tipo Encrypted para dados sensíveis
  • Auditável via repositório do DataStage

🌐 DSenv

  • Configuração de infraestrutura (libs, paths)
  • Senhas e hosts gerenciados pelo sistema
  • Variáveis usadas por múltiplos projetos
  • Ajustes de runtime do Parallel Engine
  • Variáveis de sistema operacional (NLS, LD_LIBRARY)
  • Valores que não devem aparecer no Designer
  • Gerenciado pelo sysadmin / vault externo
✅ Melhor prática: use Parameter Set para tudo que os desenvolvedores precisam ver e versionar; use DSenv para segredos e configurações de infraestrutura gerenciadas pela equipe de plataforma.

Exemplo 1 — Parameter Set para conexão com banco

1. Criar Parameter Set "CONN_DW" no Designer

ParâmetroTipoDefault
DW_HOSTString$DW_HOST (do DSenv)
DW_PORTInteger5432
DW_SCHEMAStringDW_HML
DW_USERStringetl_user
DW_PASSEncrypted{encrypted}

2. Value File — hml.apt

DW_HOST=dw-hml.empresa.com.br
DW_PORT=5432
DW_SCHEMA=DW_HML
DW_USER=etl_hml
DW_PASS={encrypted_hash_aqui}

3. Value File — prod.apt

DW_HOST=$DW_HOST  # lido do DSenv em prod
DW_PORT=5432
DW_SCHEMA=DW_PROD
DW_USER=etl_prod
DW_PASS=$DW_PASS_PROD  # senha vem do DSenv (vault)

Exemplo 2 — Invocar job via linha de comando com override

# Execução normal (usa o Value File padrão do job)
dsjob -run -jobstatus MeuProjeto JOB_CARGA_FATO

# Com Value File específico
dsjob -run -jobstatus MeuProjeto JOB_CARGA_FATO \
  -paramfile /data/params/prod.apt

# Override de parâmetro individual
dsjob -run -jobstatus MeuProjeto JOB_CARGA_FATO \
  -param DW_SCHEMA=DW_PROD_BACKUP \
  -param BATCH_SIZE=10000

# Via API REST (CP4D / DataStage as a Service)
POST /v3/jobs/{job_id}/runs
{
  "job_run_config": {
    "job_runtime": {
      "env_variables": {
        "DW_SCHEMA": "DW_PROD",
        "BATCH_SIZE": "50000"
      }
    }
  }
}

Exemplo 3 — Sequencer passando parâmetros para jobs filhos

-- No job sequenciador, defina parâmetros no nível do sequencer:
-- Parâmetro: DT_REF (String, Default: hoje)

-- Na activity de invocação do job filho, mapeie:
Job Parameter:  DT_REF  =  #DT_REF#   (herda do sequencer)
Job Parameter:  SCHEMA  =  #DW_SCHEMA#

-- DSenv complementa automaticamente variáveis não mapeadas
✅ Prefira sempre passar parâmetros explicitamente no sequencer. Dependência implícita de DSenv em jobs filhos dificulta a rastreabilidade.
